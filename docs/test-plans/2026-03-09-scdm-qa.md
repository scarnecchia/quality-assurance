# SCDM-QA Human Test Plan

## Prerequisites
- Python environment with `scdm-qa` installed (e.g. `pip install -e .`)
- All automated tests passing: `pytest tests/ -v`
- Real SCDM Parquet data available
- A `config.toml` pointing at the real data files (template below)

```toml
[tables]
demographic = "/path/to/demographic.parquet"
encounter = "/path/to/encounter.parquet"
enrollment = "/path/to/enrollment.parquet"
diagnosis = "/path/to/diagnosis.parquet"
dispensing = "/path/to/dispensing.parquet"
procedure = "/path/to/procedure.parquet"
facility = "/path/to/facility.parquet"
provider = "/path/to/provider.parquet"
death = "/path/to/death.parquet"

[options]
output_dir = "./qa-reports"
chunk_size = 50000
```

## Phase 1: CLI Smoke Test with Real Data

| Step | Action | Expected |
|------|--------|----------|
| 1 | Run `scdm-qa run config.toml` | Process completes; exit code is 0, 1, or 2 depending on data quality. Console output shows per-table progress. |
| 2 | Check `./qa-reports/` directory contents | One `{table}.html` per configured table plus `index.html`. |
| 3 | Run `scdm-qa run config.toml --table demographic` | Only `demographic.html` and `index.html` produced. No other table reports. |
| 4 | Run `scdm-qa profile config.toml` | Exit code 0. Console output shows "profiled" for each table. No validation failures reported. |
| 5 | Run `scdm-qa schema` | Lists 19 SCDM tables with column counts. |
| 6 | Run `scdm-qa schema demographic` | Shows PatID, Birth_Date, Sex, Hispanic, Race columns with types and constraints. |

## Phase 2: HTTP Server Verification

| Step | Action | Expected |
|------|--------|----------|
| 1 | Run `scdm-qa serve ./qa-reports/` | Terminal shows "Serving reports at http://localhost:8080/". Browser auto-opens. |
| 2 | Navigate to `http://localhost:8080/` in browser | Index page renders with links to all table reports. PASS/FAIL badges visible. |
| 3 | Click a table link (e.g., `demographic.html`) | Table report loads. Validation summary table visible. Profiling section visible. |
| 4 | Press Ctrl+C in terminal | Server shuts down cleanly. |
| 5 | Run `scdm-qa serve ./nonexistent/` | Exit code 2. Error message about missing directory. |

## Phase 3: Report Visual Inspection

| Step | Action | Expected |
|------|--------|----------|
| 1 | Open a table report with all checks passing | All pass rate cells show green background (#d4edda). |
| 2 | Open a table report with some failing checks | Steps with pass rate >= 99% are green, 95-99% are yellow (#fff3cd), < 95% are red (#f8d7da). |
| 3 | Look for a "Failing Rows" section in a report with failures | Collapsible `<details>` section visible. Click to expand. Failing row data displayed. |
| 4 | Click "Download CSV" button on a failing rows section | Browser downloads a CSV file. Contents match the failing rows displayed. |
| 5 | Open `index.html` | All configured tables listed with links. Tables with zero failures show "PASS". Tables with failures show "FAIL". |

## Phase 4: Memory Behaviour (AC5.1)

| Step | Action | Expected |
|------|--------|----------|
| 1 | Identify the largest Parquet file in the real data set. Note its size. | Record file size (e.g., 200MB, 1GB). |
| 2 | Run `scdm-qa run config.toml` with only that large file configured, `chunk_size = 50000`. Monitor with `top` or `/usr/bin/time -l`. | Peak RSS stays proportional to chunk_size, not file size. For a 1GB file at chunk_size=50000, RSS should stay well under 500MB. |
| 3 | Re-run with `chunk_size = 500000` (10x). | Peak RSS increases roughly proportionally to chunk size change, not to file size. |

## Phase 5: SAS File Verification (AC5.2)

| Step | Action | Expected |
|------|--------|----------|
| 1 | Obtain a real `.sas7bdat` file from an existing SCDM dataset. | File available at a known path. |
| 2 | Create a `config.toml` referencing the SAS file for its table key. | Config valid. |
| 3 | Run `scdm-qa run config.toml`. | SasReader processes the file in chunks. Validation report produced. No crash. |
| 4 | Check logs for chunk iteration messages. | Log output shows multiple chunks processed (if file is large enough). |

## End-to-End: Full Pipeline with Mixed Pass/Fail Tables

1. Create a config with 3+ tables from the real data set.
2. For one table, intentionally introduce bad data (e.g., inject nulls into a non-nullable column).
3. Run `scdm-qa run config.toml`.
4. Verify exit code is 1 or 2 (depending on failure rate vs threshold).
5. Open `index.html` — the corrupted table should show "FAIL", clean tables should show "PASS".
6. Open the failing table's report — specific column/step failures visible with red highlighting.
7. Open a passing table's report — all green.

## End-to-End: Custom Rules Extension

1. Create directory `custom_rules/`.
2. Create `custom_rules/demographic_rules.py` with:
   ```python
   def extend_validation(validation, data):
       return validation.col_vals_not_null(columns="Birth_Date")
   ```
3. Add `custom_rules_dir = "./custom_rules"` to `[options]` in config.toml.
4. Run `scdm-qa run config.toml --table demographic`.
5. Verify the report shows the extra Birth_Date not-null check as an additional validation step.
