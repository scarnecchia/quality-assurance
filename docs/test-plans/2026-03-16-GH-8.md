# Human Test Plan: Single-Pass Chunk-Consumer Architecture (GH-8)

## Prerequisites
- Environment: local dev with `uv` installed, Python 3.12+
- `uv run pytest` passes (470 passed, 9 skipped for SAS is acceptable)
- `pyreadstat` installed if verifying SAS-specific steps (optional but recommended)
- A sample TOML config pointing to at least one Parquet file and one SAS file (e.g., `demographic.parquet`, `demographic.sas7bdat`)

## Phase 1: Code Structure Verification

| Step | Action | Expected |
|------|--------|----------|
| 1.1 | Open `src/scdm_qa/pipeline.py` and locate `_process_table()` | Function exists |
| 1.2 | Count the lines of `_process_table()` (from `def` to last line before next `def`/class or EOF) | <= 30 lines |
| 1.3 | Verify `_process_table()` contains no inline DuckDB connection management (`duckdb.connect`, `CREATE VIEW`, SQL strings) | No DuckDB code present in the function body |
| 1.4 | Verify `_process_table()` contains no chunk iteration (`for chunk in`, `reader.chunks()`) | No chunk loop present |
| 1.5 | Verify `_process_table()` contains no SAS-vs-Parquet branching (`if ... .sas7bdat`, `if is_sas`) | No format branching present |
| 1.6 | Verify `_process_table()` delegates to `TableValidator` (look for `TableValidator(...)` construction and `.run()` call) | Delegation is the primary logic path |

## Phase 2: Chunk Ordering Barrier Inspection

| Step | Action | Expected |
|------|--------|----------|
| 2.1 | Open `src/scdm_qa/validation/table_validator.py` and locate `_broadcast_chunks()` | Method exists |
| 2.2 | Identify the `ThreadPoolExecutor` submit calls for each accumulator per chunk | Each accumulator is submitted as a future for the current chunk |
| 2.3 | Verify that after submitting all futures for chunk N, the code calls `.result()` (or `as_completed`) on ALL futures before proceeding to chunk N+1 | Barrier pattern present -- no chunk N+1 dispatch until all N futures resolve |
| 2.4 | Verify that if any future raises, the exception is re-raised (not caught/logged/swallowed) | Exception propagation is explicit |

## End-to-End: Full Refactor Round-Trip

**Purpose:** Validate that the entire pipeline produces identical output pre- vs post-refactor for a real dataset.

**Steps:**
1. Check out the commit before the refactor (`git stash` current changes or use a second worktree at the base SHA `7afc9b9`)
2. Run `uv run scdm-qa run <config.toml>` against a representative Parquet dataset and save the output directory
3. Check out the refactored branch (`GH-8` at HEAD)
4. Run the same command with the same config and dataset
5. Diff the two output directories (excluding timestamps/metadata): compare `validation_result.json`, profiling stats, check counts, exit code
6. All values should be identical (SAS outputs may have additional global check entries -- that is expected and additive)
