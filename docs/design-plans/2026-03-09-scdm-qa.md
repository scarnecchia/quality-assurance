# SCDM Quality Assurance Tool Design

## Summary

`scdm-qa` is a Python CLI and library that validates health data tables conforming to the Sentinel Common Data Model (SCDM) against a machine-readable spec (`tables_documentation.json`). It handles Parquet and SAS7BDAT input files, runs validation rules auto-generated from the spec, and produces self-contained interactive HTML reports that analysts can explore locally without any server infrastructure.

The core engineering challenge is validating datasets that can reach terabyte scale without loading them entirely into memory. The tool solves this via a chunked pipeline: data is read in bounded batches, validated per-chunk using pointblank, and results are accumulated and merged into a single report per table. Checks that inherently require a global view — uniqueness constraints and sort order — are handled separately via DuckDB (when available) or in-memory key accumulation. Profiling stats (completeness, value distributions, date ranges, cardinality) are collected in the same pass as validation to avoid a second scan.

## Definition of Done

1. **A Python CLI + library** (`scdm-qa` or similar) that validates SCDM data in Parquet or SAS format against rules derived from `tables_documentation.json`. Installable via pip/pyproject.toml, uv-compatible.

2. **Validation rules cover the full SCDM spec** — type checks, nullability, allowed values, sort order, uniqueness constraints, conditional rules (e.g., DDate required when EncType=IP), and string length checks — all auto-generated or inferred from the JSON spec.

3. **Basic data profiling** — completeness rates, value distributions, date range reasonableness, cardinality counts — included alongside spec validation.

4. **Interactive HTML dashboard** (via pointblank) that analysts can explore locally — pass/fail summaries, threshold tolerance, drill-down into failures.

5. **Handles TB-scale data** via Polars lazy scanning (Parquet) and chunked reading (SAS), with DuckDB available as an optional engine.

6. **Adding new validation rules is easy** — pointblank's step-by-step API, with rules defined declaratively against the SCDM schema definitions.

## Glossary

- **SCDM (Sentinel Common Data Model)**: A standardised schema for health claims data used across the Sentinel System, a U.S. FDA programme. Defines 19 tables (e.g., Enrollment, Demographic, Encounter) with specific column types, nullability rules, allowed values, and sort/uniqueness constraints.
- **`tables_documentation.json`**: A machine-readable JSON file that encodes the full SCDM table specifications — column names, types, enumerations, sort order, uniqueness keys, and conditional nullability rules. Used as the single source of truth for schema codegen.
- **Schema codegen**: Code generation — the process of automatically producing Python source files (one per SCDM table) from `tables_documentation.json` rather than writing them by hand. Ensures schema definitions stay in sync with the spec.
- **pointblank**: A Python data validation library (ported from the R package of the same name) that provides a step-by-step API for defining validation rules and generating interactive HTML reports with pass/fail summaries.
- **Polars**: A Rust-backed DataFrame library for Python. Used here for its lazy evaluation (`scan_parquet()`) and memory-efficient row-group batching on Parquet files.
- **DuckDB**: An in-process analytical SQL engine. Used as an optional dependency for global uniqueness checks (`GROUP BY ... HAVING COUNT(*) > 1`) on Parquet files, where it can query without full materialisation.
- **pyreadstat**: A Python library for reading SAS7BDAT (and other statistical file formats) with support for chunked reading via `read_file_in_chunks()`, avoiding full in-memory loading.
- **great_tables**: A Python library for producing styled, publication-quality tables in HTML. Used here to render the profiling summary section of each report.
- **SAS7BDAT**: The binary file format produced by SAS statistical software. Common in clinical and administrative health data environments.
- **Parquet**: A columnar binary file format optimised for analytical workloads. Supports row-group-level access, which enables bounded memory reads without loading entire files.
- **Row group**: A horizontal partition within a Parquet file. Polars can scan individual row groups, making them the natural unit of chunked reading.
- **Chunked pipeline**: The architecture pattern where data is read and processed in fixed-size batches rather than all at once, keeping peak memory bounded regardless of input size.
- **`ValidationAccumulator`**: The internal component that merges per-chunk pointblank results — summing pass/fail counts and collecting a bounded sample of failing rows — into a single aggregate result for reporting.
- **Conditional nullability rule**: A validation rule where a field is required only under certain conditions — e.g., `DDate` (discharge date) must be non-null when `EncType` (encounter type) is `IP` (inpatient) or `IS` (inpatient stay).
- **typer**: A Python CLI framework built on top of Click. Used here for the `scdm-qa` command and its subcommands (`run`, `profile`, `schema`, `serve`).
- **structlog**: A Python structured logging library. Produces both human-readable console output and machine-readable JSON log files.
- **`TableReader` protocol**: A Python `Protocol`-based interface that both the Parquet and SAS readers implement, allowing the rest of the pipeline to be format-agnostic.
- **Cardinality**: The number of distinct values in a column. High cardinality on an identifier column (e.g., PatID) is expected; high cardinality on an enum column would be a data quality signal.
- **Threshold tolerance**: A configurable pass/fail threshold in pointblank — e.g., a check passes if ≥ 99% of rows conform, rather than requiring 100%. Reflected in the HTML report's colour coding and the CLI exit code.

## Architecture

Chunked validation pipeline with merged reporting. Data is read in bounded chunks, validated per-chunk via pointblank, and results aggregated into a single HTML report per table.

```
Input (Parquet/SAS)
     │
Chunked Reader
  ├── Parquet: Polars scan_parquet() → collect batches by row group
  └── SAS: pyreadstat read_file_in_chunks() → convert to Polars
     │
Per-Chunk Validation (pointblank)
  → nullability, enum membership, string lengths,
    type conformance, conditional rules
     │
Results Accumulator
  → sums pass/fail counts across chunks
  → captures bounded failing row extracts
     │
Global Validation (DuckDB, optional)
  → uniqueness constraints (GROUP BY + HAVING)
  → sort order (cross-chunk boundary checks)
     │
Profiling (Polars/DuckDB aggregation)
  → completeness rates, value distributions,
    date range min/max, cardinality counts
     │
Report Builder
  ├── pointblank HTML report (validation results)
  └── great_tables profiling summary
     │
Self-contained HTML output
  ├── per-table report files
  └── index page linking all tables
```

**Schema Layer** — Python schema definitions code-generated from `tables_documentation.json` via `scripts/generate_schemas.py`. One module per SCDM table under `src/scdm_qa/schemas/`. Each module defines column names, types, nullability, allowed values, uniqueness constraints, sort order, and conditional rules. A schema registry maps table names to schema modules.

Each schema module exposes a `build_validation(data) -> pb.Validate` function that constructs a pointblank Validate chain from the schema definition. Custom user rules can be appended to this chain via a Python extension file per table.

**Reader Layer** — Two readers behind a common `TableReader` protocol:

```python
class TableReader(Protocol):
    def metadata(self) -> TableMetadata: ...
    def chunks(self) -> Iterator[pl.DataFrame]: ...
```

Parquet reader uses Polars `scan_parquet()` for metadata extraction and iterates row groups in batches. SAS reader uses pyreadstat `read_file_in_chunks()` and converts each pandas chunk to Polars. Format detected by file extension. Chunk size configurable (default 500k rows).

**Validation Layer** — Two tiers:

1. Per-chunk: pointblank `Validate().interrogate()` runs against each chunk. Results accumulated in a `ValidationAccumulator` that sums counts across chunks and captures bounded failing row extracts.

2. Global: checks requiring the full dataset. Uniqueness via DuckDB `SELECT ... GROUP BY {key_cols} HAVING COUNT(*) > 1` on Parquet, or via accumulated key sets for SAS (falling back to DuckDB temp table if set exceeds memory threshold). Sort order verified by checking cross-chunk boundaries (last value of chunk N ≤ first value of chunk N+1).

**Reporting Layer** — `ValidationAccumulator` builds a synthetic pointblank `Validate` object from aggregated results and calls `get_tabular_report()` for the interactive HTML report. Profiling summary generated as a companion section using great_tables. Output is self-contained HTML files — one per table plus an index. Optional `--serve` flag for local HTTP browsing.

**Configuration** — TOML file:

```toml
[tables]
enrollment = "/data/enrollment.parquet"
demographic = "/data/demographic.sas7bdat"
encounter = "/data/encounter.parquet"

[options]
chunk_size = 500000
output_dir = "./qa-reports"
```

## Existing Patterns

Investigation of the scdm-to-parquet project (`/Users/scarndp/dev/Sentinel/scdm-to-parquet/`) found:

- **Schema codegen from `tables_documentation.json`** — same JSON spec, different output targets (Arrow types there, validation rules here). Codegen script can share the JSON parsing logic.
- **pyreadstat chunked reading** via `read_file_in_chunks()` — proven pattern, adopted here for SAS input.
- **typer CLI** — adopted here with similar subcommand structure.
- **structlog for logging** — adopted here.
- **TOML configuration** — same table-name-to-file-path mapping structure.
- **Per-table isolation** — each table validates independently; a failure in one does not block others.

**Divergences from scdm-to-parquet:**
- Polars as primary data engine (scdm-to-parquet uses pandas → Arrow directly)
- pointblank for validation (scdm-to-parquet does file-level checks only)
- DuckDB as optional dependency for global checks (scdm-to-parquet doesn't use DuckDB)
- HTML report output (scdm-to-parquet produces Parquet files + log summaries)
- great_tables for profiling summary display

## Additional Considerations

**Conditional rule parsing:** The `missing_allowed` field in the JSON spec uses free-text descriptions for conditional rules (e.g., "Yes, conditional on EncType value"). The codegen script must parse these into structured conditional checks. Cases that can't be parsed automatically should be flagged for manual rule definition.

**DuckDB as optional dependency:** Global checks (uniqueness, sort order on large Parquet) benefit from DuckDB's lazy scanning. When DuckDB isn't installed, uniqueness falls back to in-memory key accumulation (works for moderate-scale data) and sort order uses cross-chunk boundary checks only. The `[duckdb]` optional dependency group makes this explicit: `pip install scdm-qa[duckdb]`.

**Custom rule extensibility:** Users can provide a Python file per table with additional pointblank validation steps. The tool loads these and appends them to the generated chain. This supports site-specific rules beyond the SCDM spec without modifying generated code.

**No row-level data in logs or reports beyond bounded extracts.** Failing row extracts are capped (default 500 rows per check) and only appear in the HTML report, not in log files. This aligns with the scdm-to-parquet project's data handling approach.
