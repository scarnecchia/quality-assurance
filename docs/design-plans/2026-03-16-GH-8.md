# Single-Pass Chunk-Consumer Architecture Design

## Summary

The current L1 per-table validation pipeline is implemented as a monolithic `_process_table()` function in `pipeline.py` that directly manages chunk iteration, accumulator calls, and DuckDB global checks. This design makes it hard to extend (adding a new accumulator requires touching the read loop) and leaves SAS files unable to participate in DuckDB-based global checks. GH-8 refactors this into a `TableValidator` class with a formal `ChunkAccumulator` protocol: any object that implements `add_chunk()` and `result()` can be registered and will receive every chunk without any changes to the read loop itself.

The implementation proceeds in five phases: establishing the protocol and extracting the SAS-to-Parquet conversion utility to a shared location; wrapping the existing per-chunk validation pipeline behind the protocol; building `TableValidator` with concurrent chunk broadcasting via `ThreadPoolExecutor`; rewiring `pipeline.py` to delegate to `TableValidator`; and cleaning up the dead code that results. Throughout, the concurrency model keeps things tractable — accumulators run in parallel on a given chunk, but the next chunk is held until all accumulators finish the current one, so no accumulator needs to be thread-safe against itself.

## Definition of Done

1. **TableValidator class** encapsulates the full L1 lifecycle: reader creation, chunk broadcasting, DuckDB global checks, and result assembly. `_process_table()` becomes a thin orchestrator that delegates to `TableValidator`.

2. **ChunkAccumulator protocol** defines a shared interface (e.g., `add_chunk()`, `result()`) that `ValidationAccumulator`, `ProfilingAccumulator`, and any future per-chunk checks implement. New accumulators can be registered without modifying the read loop.

3. **Async-capable broadcast** — the chunk broadcast mechanism supports concurrent accumulator processing (e.g., via thread/task pool), so accumulators can process chunks in parallel.

4. **SAS global checks enabled** — `TableValidator` uses GH-6's streaming SAS-to-Parquet conversion to produce a temp Parquet file, enabling DuckDB global checks on SAS inputs (no longer skipped).

5. **No regression** in validation results, profiling output, or reporting.

## Glossary

- **ChunkAccumulator**: A `typing.Protocol` defining the shared interface (`add_chunk()`, `result()`) that all per-chunk consumers implement. Enables registration of new consumers without modifying the read loop.
- **TableValidator**: The new class encapsulating the full L1 lifecycle — reader creation, chunk broadcasting, DuckDB global checks, and result assembly. Replaces the monolithic `_process_table()` function.
- **ValidationChunkAccumulator**: An adapter class that wraps the existing `build_validation → interrogate → accumulate` pipeline behind the `ChunkAccumulator` protocol.
- **ProfilingAccumulator**: The existing class that accumulates streaming column statistics. Already conforms to the `ChunkAccumulator` shape; no changes needed.
- **ValidationAccumulator**: The existing class that collects per-chunk validation step results into a combined `ValidationResult`.
- **L1 / L2**: The two validation phases in the pipeline. L1 is per-table (chunk-based checks plus DuckDB global checks). L2 is cross-table (referential integrity, etc.).
- **DuckDB global checks**: Validation rules (uniqueness, sort order, etc.) that require the full table to be queryable at once. Implemented via an in-process DuckDB connection reading a Parquet file.
- **`converted_parquet()` context manager**: A context manager in `readers/conversion.py` that streams a SAS file to a temp Parquet file and deletes it on exit, enabling DuckDB global checks on SAS inputs.
- **`ThreadPoolExecutor`**: A Python standard-library executor that runs callables in a thread pool. Used here to dispatch each chunk to all accumulators concurrently.
- **`typing.Protocol`**: A Python structural-subtyping mechanism. A class satisfies a Protocol if it has the required methods — no explicit inheritance needed. Verified at runtime via `isinstance()` when the Protocol is `runtime_checkable`.
- **`runtime_checkable`**: A decorator applied to a `typing.Protocol` that enables `isinstance()` checks against it at runtime, used in tests to verify conformance.
- **GIL (Global Interpreter Lock)**: CPython's lock that normally limits true thread parallelism. Polars operations are backed by Rust and release the GIL, so `ThreadPoolExecutor` achieves real parallelism for the heavy DataFrame work.
- **`StepResult`**: The internal data model for a single validation check outcome, carrying `check_id`, `severity` ("Fail" | "Warn" | "Note" | `None`), pass/fail counts, and optional failing rows.
- **pointblank**: The Python library used to express and evaluate validation rules declaratively against DataFrames.
- **Polars**: The DataFrame library used throughout the pipeline. All chunks are `polars.DataFrame` instances.
- **SAS7BDAT**: A proprietary SAS binary data format. Read via `pyreadstat` in chunked streaming mode.
- **Fan-out / broadcast**: The pattern where a single chunk is dispatched to multiple consumers (accumulators) simultaneously.

## Architecture

Extract the L1 per-table lifecycle from `pipeline._process_table()` into a `TableValidator` class with a formalized accumulator protocol. The architecture has two distinct phases:

**Phase A — Chunk-based pass:** A single `TableReader` yields chunks. `TableValidator` broadcasts each chunk to all registered `ChunkAccumulator` instances via `ThreadPoolExecutor`. Accumulators process chunks concurrently (one thread per accumulator), with a synchronization barrier between chunks to preserve ordering.

**Phase B — DuckDB global checks:** After the chunk pass completes, `TableValidator` resolves a Parquet path (native Parquet files used directly; SAS files converted via `readers/conversion.py`) and runs all DuckDB-based global checks against it. This is unchanged from GH-7's implementation — just relocated into `TableValidator`.

**Key components:**

- `ChunkAccumulator` — `typing.Protocol` with `add_chunk(chunk: pl.DataFrame) -> None` and `result() -> Any`. Thread-safety contract: each accumulator owns its own mutable state exclusively.
- `TableValidator` — Owns the reader, broadcasts chunks, runs global checks, assembles combined results. Accumulators are injected at construction.
- `ValidationChunkAccumulator` — Adapter wrapping the existing `build_validation() → interrogate() → ValidationAccumulator` pipeline behind the `ChunkAccumulator` protocol.
- `ProfilingAccumulator` — Already conforms to `ChunkAccumulator` (has `add_chunk()` + `result()`). No changes needed.

**Concurrency model:** `ThreadPoolExecutor` with one worker per accumulator. Polars releases the GIL during Rust-backed DataFrame operations, so threads achieve real parallelism for the heavy lifting. The `concurrent.futures` API allows swapping to `ProcessPoolExecutor` later if GIL contention is measured — same interface, different executor.

**SAS file handling:** `_convert_sas_to_parquet()` moves from `validation/cross_table.py` to `readers/conversion.py` as a public function with a `converted_parquet()` context manager for automatic temp file cleanup. Both `TableValidator` and `run_cross_table_checks()` import from this shared location.

## Existing Patterns

Investigation found two relevant patterns in the current codebase:

**Accumulator pattern (informal):** `ValidationAccumulator` and `ProfilingAccumulator` both follow `add_chunk()` / `result()` but without a shared protocol. This design formalizes the pattern with `ChunkAccumulator` as a `typing.Protocol`, making it explicit and extensible.

**Reader protocol:** `TableReader` in `readers/base.py` defines `metadata()` and `chunks()` via `typing.Protocol`. `ChunkAccumulator` follows the same style — `runtime_checkable` Protocol, no ABC inheritance required.

**Divergence from existing patterns:**

- `run_validation()` in `validation/runner.py` currently owns both the chunk loop and the validation-per-chunk logic. This design splits those responsibilities: `TableValidator` owns the loop, `ValidationChunkAccumulator` owns the per-chunk validation. `run_validation()` becomes obsolete and can be removed once migration is complete.

- `_process_table()` currently manages DuckDB connections inline. This design moves connection lifecycle into `TableValidator._run_global_checks()`.

## Additional Considerations

**Thread safety:** The `ThreadPoolExecutor` broadcast waits for all accumulators to finish chunk N before dispatching chunk N+1. This means accumulators never process multiple chunks concurrently — concurrency is *between* accumulators on the same chunk. This simplifies thread safety: each accumulator only needs to avoid sharing state with *other* accumulators, not with itself across chunks.

**Error propagation:** If any accumulator raises during `add_chunk()`, `future.result()` re-raises in the main thread. `TableValidator` should let this propagate — fail fast rather than silently dropping accumulator errors.

**SAS global checks — new capability:** Enabling DuckDB global checks on SAS inputs is a net-new feature. Some global checks may surface findings that were previously invisible for SAS-only datasets. This is expected and correct behaviour, but worth noting in release notes.
