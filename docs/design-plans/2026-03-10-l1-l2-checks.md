# L1 & L2 Validation Checks Design

## Summary

This document covers the implementation of L1 and L2 validation checks derived from the legacy SAS QA package used in Sentinel Common Data Model (SCDM) processing. The goal is to bring parity between the existing Python tool and a reference SAS implementation — specifically 11 new checks identified by their SAS CheckIDs — so that outputs from both systems can be cross-referenced and reconciled.

L1 checks are column-level data quality rules (e.g., leading whitespace, unexpected zeros, entirely unpopulated columns) that run inside the existing per-chunk pipeline loop via pointblank assertions. L2 checks are cross-record rules that require a global view of the table — things like overlapping enrollment spans, date ordering violations, and invalid encounter field combinations — and are implemented as new functions in the global checks module, called after the per-chunk loop completes. Both check types emit `StepResult` objects enhanced with a new `check_id` field that maps back to the originating SAS CheckID, enabling direct traceability between tool outputs.

## Definition of Done

1. **L1 per-chunk checks implemented** — CheckID 111 (variable not populated), 122 (leading spaces), 124 (unexpected zeros), 128 (non-numeric characters) all produce validation results with correct pass/fail counts.

2. **L2 single-table checks implemented** — CheckID 226 (date ordering), 215 (overlapping enrollment spans), 216 (consecutive non-bridged enrollment), 236 (missing underlying cause of death), 237 (multiple underlying causes of death), 244 (invalid ENC field combination), 245 (invalid ENC combo rate threshold) all produce validation results.

3. **SAS CheckID cross-referencing** — Every new check carries its SAS CheckID in StepResult, enabling traceability between this tool's output and the legacy SAS QA package.

4. **Severity levels respected** — Checks marked as Fail in the SAS reference produce errors; checks marked as Warn produce warnings; checks marked as Note produce informational results.

5. **Existing checks unaffected** — All pre-existing validation rules continue to work identically. StepResult's new check_id field defaults to None for existing checks.

6. **Test coverage** — Each check has at least one happy-path and one failure-path test using synthetic data.

## Glossary

- **SCDM (Sentinel Common Data Model)**: A standardised data structure used by the FDA Sentinel System for analysing distributed health data. Tables like ENC, ENR, DEM, COD, etc. are SCDM table abbreviations.
- **L1 check**: A per-chunk, column-level data quality check (e.g., leading spaces, unexpected zeros). Runs inside the existing validation loop without architectural changes.
- **L2 check**: A global, cross-record data quality check that requires a full table scan (e.g., overlapping enrollment spans, date ordering). Runs after the per-chunk loop completes.
- **SAS CheckID**: A numeric identifier from the legacy SAS QA reference package (e.g., CheckID 226). Used as a cross-reference key between this tool's output and the SAS equivalent.
- **StepResult**: A frozen dataclass that holds the outcome of a single validation rule — pass/fail counts, sample failing rows, and (new) the SAS `check_id`. The atomic unit of validation output.
- **pointblank**: A Python data validation library used to declare and run assertions against DataFrames. Assertions like `col_vals_regex` and `col_vals_not_null` are pointblank API calls.
- **`build_validation()`**: The function in `schemas/validation.py` that constructs a pointblank `Validate` object for a given table schema. L1 checks 122, 124, and 128 are added here.
- **`ValidationAccumulator`**: The class in `validation/accumulator.py` that merges per-chunk `StepResult` objects into a single coherent result across all chunks of a table.
- **global checks**: Validation logic that cannot be evaluated per-chunk and instead requires accumulating data across the entire table (uniqueness, sort order, all L2 checks).
- **`L1CheckDef`**: A new frozen dataclass (added in Phase 1) that defines a single L1 check — which table, which column, which check type, and what severity.
- **check registry (`checks.py`)**: A new module holding hardcoded `L1CheckDef` instances derived from the SAS lookup tables. Single source of truth for check-to-table-to-column mappings.
- **DuckDB fast path**: An optional SQL-based execution path for global checks that queries Parquet files directly via DuckDB, falling back to in-memory Polars when DuckDB is unavailable.
- **`lkp_all_flags` / `lkp_all_l1`**: SAS lookup tables from the reference QA package that define which columns on which tables receive which L1 checks and at what severity.
- **`lkp_enc_l2`**: A 96-row SAS lookup table defining the valid combinations of EncType, DDate state, Discharge_Disposition, and Discharge_Status. Used as the anti-join matrix for checks 244 and 245.
- **`lkp_rate_threshold`**: A SAS lookup table defining percentage thresholds for check 245 (invalid ENC combo rate by EncType). Three threshold tests (TestIDs 90, 91, 92).
- **CauseType='U'**: A field value in the COD (Cause of Death) table designating the underlying cause of death. Checks 236 and 237 validate that each patient has exactly one such record.
- **anti-join**: A relational operation that returns rows from one dataset that have no match in another. Used in check 244 to identify ENC rows not matching any valid combination in the `lkp_enc_l2` matrix.
- **na_pass**: A pointblank assertion parameter that causes null values to be treated as passing rather than failing. Used in L1 checks 122, 124, and 128.
- **special missing (`.S` in SAS)**: A SAS-specific encoding for a missing value with semantic meaning (distinct from a plain null). Has no native Polars equivalent; requires a sentinel value or separate boolean column.
- **severity levels (Fail / Warn / Note)**: The three outcome levels from the SAS reference, mapped to error, warning, and informational results respectively.

## Architecture

Extends the existing chunked validation pipeline with two new check categories:

**L1 checks** (per-chunk) — New pointblank assertions added to `build_validation()` in `src/scdm_qa/schemas/validation.py`. These run inside the existing per-chunk validation loop with no architectural change. Check 111 (variable not populated) is the exception — it requires a global view and runs as a separate scan after chunked validation.

**L2 checks** (global, single-table) — New functions in `src/scdm_qa/validation/global_checks.py`, called from `pipeline.py` after existing global checks (uniqueness, sort order). Each L2 check creates a fresh reader for its scan and returns `StepResult` objects that are appended to the validation result.

**Check registry** — A new module `src/scdm_qa/schemas/checks.py` holds hardcoded `L1CheckDef` instances derived from the SAS lookup tables (`lkp_all_flags`, `lkp_all_l1`). These define which columns on which tables get which checks, with severity levels. The registry is the single source of truth for check-to-table-to-column mappings.

**StepResult enhancement** — `check_id: str | None` field added to `StepResult`, propagated through `ValidationAccumulator` and `_build_step_descriptions()`. Existing checks get `None`; new checks carry their SAS CheckID string.

```
Existing Pipeline (unchanged)
     │
Per-Chunk Validation
  → existing: nullability, enum, length, conditional
  → NEW: L1 checks 122, 124, 128 (pointblank assertions)
     │
Results Accumulator (enhanced with check_id)
     │
Global Checks
  → existing: uniqueness, sort order
  → NEW: L1 check 111 (not populated)
  → NEW: L2 checks 226, 215, 216, 236, 237, 244, 245
     │
Report (unchanged — just renders more steps)
```

## Existing Patterns

This design follows established patterns from the codebase:

- **Per-chunk pointblank assertions** — L1 checks 122/124/128 follow the same pattern as existing `col_vals_not_null`, `col_vals_in_set`, and `col_vals_regex` calls in `build_validation()`. The `_build_step_descriptions()` mirror is maintained.

- **Global check functions** — L2 checks follow the `check_uniqueness()` and `check_sort_order()` pattern in `global_checks.py`: take a reader, iterate chunks, return `StepResult` objects. Pipeline wires them in after existing global checks.

- **DuckDB fast path** — Check 215 (overlapping spans) uses DuckDB self-join for Parquet when available, with polars fallback. Matches the existing `_uniqueness_duckdb()` / `_uniqueness_in_memory()` pattern.

- **Frozen dataclasses** — `L1CheckDef` follows the existing `ColumnDef`, `ConditionalRule`, `TableSchema` pattern.

**No divergences from existing architecture.** All new checks integrate into the existing pipeline flow.

### Phase 6: L2 Enrollment Checks (215, 216)
**Goal:** Detect overlapping enrollment spans and consecutive non-bridged enrollment gaps.

**Components:**
- `src/scdm_qa/validation/global_checks.py` — New `check_overlapping_spans()` and `check_enrollment_gaps()` functions
- `src/scdm_qa/pipeline.py` — Wire into `_process_table()` for enrollment table

Logic for 215 (overlapping spans):
- Collect all (PatID, Enr_Start, Enr_End) tuples, sort by PatID + Enr_Start

- DuckDB fast path: self-join SQL for Parquet; polars fallback

Logic for 216 (enrollment gaps):
- Same sorted data. Consecutive spans where previous Enr_End + 1 day < current Enr_Start → gap
- Warn severity

**Dependencies:** Phase 1

## Additional Considerations

**Check 111 severity mapping:** The SAS reference distinguishes between Note (informational — ImputedHispanic, ImputedRace, PlanType, PayerType) and Fail (critical — PDX, DDate, Discharge_Disposition, Discharge_Status, Admitting_Source). The severity should map to pointblank thresholds or a separate severity field on StepResult.

**Check 245 threshold values:** The SAS reference defines three TestIDs (90, 91, 92) for rate thresholds but the exact threshold percentages need to be confirmed from the `lkp_rate_threshold` lookup table. If unavailable, sensible defaults should be documented.

**DDate special missing state:** Check 244 requires distinguishing three DDate states: Null, Present (has a date value), and Special missing (.S in SAS). In Polars, special missing doesn't exist as a concept — this will need an encoding convention (e.g., a sentinel date value or a separate boolean column). The implementation plan should address this mapping.

**No second design plan needed.** All 8 phases fit within the writing-plans limit.
